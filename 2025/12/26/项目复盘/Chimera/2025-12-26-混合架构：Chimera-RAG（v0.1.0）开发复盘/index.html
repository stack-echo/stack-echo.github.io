混合架构：Chimera-RAG（v0.1.0）开发复盘

引言：在 EHS 安全合规领域，如何让 AI 真正“读懂”晦涩的法规文档？本文记录了我基于 Go (网关) + Python (大脑) 混合架构，结合 Qdrant 与 DeepSeek，构建 RAG 系统的全过程。从 gRPC 的 4MB 限制到上下文窗口的调优，复盘 v0.1.0 版本的关键技术决策与实战踩坑。

1. 缘起：当 Go 的快遇上 Python 的智

在构建一个垂直领域的 RAG (检索增强生成) 系统时，技术选型往往面临两难：

  • 全 Python 栈（如 LangChain 生态）：AI 组件开箱即用，但在处理高并发 HTTP 请求、大文件 I/O 吞吐时，性能与资源占用往往成为瓶颈。

  • 全 Go 栈：拥有极致的并发性能和极低的部署成本，但缺乏成熟的 PDF 深度解析、Agent 编排等 AI 生态。

为了兼顾性能与生态，我决定采用异构混合架构来构建 Chimera-RAG（奇美拉）。 Chimera 寓意着集各家所长的混合体——利用 Go 处理高并发网关与任务调度，利用 Python 处理复杂的文档解析与推理，两者通过 gRPC 紧密协作，取长补短。

2. 架构深度解析 (v0.1.0)

Chimera-RAG 不是一个单体应用，而是一个典型的微服务架构。我们将系统划分为四个核心层级，每一层各司其职，互不干扰。

2.1 接入层 (Gateway Layer) - Go 的主场

  • 技术栈: Go (Gin) + SSE (Server-Sent Events)

  • 职责:

    • 作为系统的“守门员”，处理所有 HTTP 请求、鉴权与流控。

    • 零拷贝上传: 接收前端的大文件流，直接Pipe到 MinIO 对象存储，最小化内存占用。

    • 长连接管理: 维持与前端的 SSE 连接，像“打字机”一样实时推送 AI 的思考过程与回答。

  • 设计思考: 这一层选 Go 是无比正确的。Goroutine 极低的切换成本，使其在处理海量并发连接和 I/O 密集型任务时，性能远超 Python 的 asyncio 或线程池方案。之后有时间会考虑做个benchmark进行精细对比。

2.2 任务调度层 (Scheduling Layer) - 异步解耦

  • 技术栈: Redis (List/PubSub) + Go Worker Pool

  • 职责:

    • 实现用户体验与繁重计算的解耦。前端上传文件后，Go 网关并不等待解析完成，而是将任务丢进 Redis 队列并立即返回“上传成功”。

    • 后台的 Go Worker 默默消费任务，负责协调数据的流动。

  • 设计思考: 如果没有这一层，解析一个 500 页的 PDF 会导致浏览器长时间转圈等待。异步化是提升用户体验的关键一步。当然，在使用时也暴露了新的问题：没有做批量上传和文档归类，会考虑在后续进行优化。

2.3 认知计算层 (Cognitive Layer) - Python 的舞台

  • 技术栈: Python + gRPC + PyMuPDF + SentenceTransformers

  • 职责:

    • 它是系统的“大脑”，负责“脏活累活”。接收 Go 传来的文件流，利用 PyMuPDF 将 PDF“嚼碎”成文本切片，再调用 Embedding 模型将其向量化。

    • 通过 gRPC Server 暴露内部能力，供 Go Worker 调用。

  • 设计思考: 选择 gRPC 而不是 HTTP RESTful，是为了利用 Protobuf 的高效序列化和 HTTP/2 的多路复用特性，确保在内部服务间传输大文本和向量数据时的高性能。

2.4 记忆与生成层 (Memory & Generation)

  • 技术栈: Qdrant + DeepSeek V3 API

  • 职责:

    • Qdrant: 存储向量化的“长期记忆”。我们在 Payload 中存储了 PageNumber 和原文片段，这是实现“引用溯源”的基础。

    • DeepSeek: 最终的“表达者”。它基于我们检索到的上下文，组织语言回答用户，并支持 Markdown 格式输出。

3. 关键技术挑战与“踩坑”实录

v0.1.0 的开发并非一帆风顺，以下是三个印象最深刻的技术卡点与解决方案。

3.1 gRPC 的“4MB”隐形墙

  • 现象：在测试上传我的硕士开题报告（约 8.5MB）时，Python gRPC Server 抛出了 ResourceExhausted 错误，拒绝接收。

  • 根因：gRPC 默认限制单次消息大小为 4MB (4,194,304 bytes)。这是一个常见的微服务通信陷阱。

  • 解决：需要在 Python Server 端和 Go Client 端同时修改配置。

    • Go Client: 在 grpc.Dial 时添加 grpc.WithDefaultCallOptions(grpc.MaxCallRecvMsgSize(100*1024*1024), ...)。

    • Python Server: 在 grpc.server 初始化时添加 options=[('grpc.max_receive_message_length', 100*1024*1024), ...]。

    • 将阈值统一提升至 100MB，以适应未来的高清扫描件需求。

3.2 “幻觉”治理与上下文窗口调优

  • 现象：起初，对于一些细节问题，DeepSeek 的回答非常空泛，甚至提示“提供的上下文信息不完整，无法回答”。

  • 根因：

    1. 检索量不足：为了省 Token，最初只检索了 TopK=3 个切片，导致关键信息遗漏。

    2. Prompt 结构单一：只喂给了模型纯文本，没有包含页码等元数据，模型无法建立空间感。

  • 解决：

    1. 扩大视野：充分利用 DeepSeek V3 的长上下文能力和低成本优势，将 TopK 提升至 15。

    2. 结构化 Prompt：重构提示词模板，显式注入结构化信息：

      这显著提升了回答的准确率和“引用感”。

3.3 极速前端：验证闭环

  • 背景：由于后端进度过快，为了验证全链路闭环，需要快速搭建一个前端界面。

  • 解决：基于 Vite + Vue 3 + Arco Design，完成了一个单页应用。利用 @microsoft/fetch-event-source 库完美处理了后端的 SSE 流式响应，配合 markdown-it 实现了类似 ChatGPT 的逐字打印渲染效果。这证明了全栈能力在快速原型验证中的重要性。

4. 成果

目前，Chimera-RAG 的 v0.1.0 版本已经具备了“真实阅读能力”。它不再是简单地根据文件名猜答案，而是能精准定位到 PDF 文档中的具体段落，并结合上下文进行推理。

5. 展望 v0.2.0

v0.1.0 的成功验证了异构架构的可行性，但也暴露了 MVP 与企业级产品之间的差距。为了构建更强大、更安全的知识库平台，v0.2.0 的技术演进路线已规划完毕：

  • 数据基座升级：

    • 引入 PostgreSQL：存储用户体系、组织架构及文件夹目录树，实现 RBAC 权限控制。

    • 引入 NebulaGraph：构建领域知识图谱，实现 GraphRAG，提升对复杂实体关系的推理能力。

  • 解析能力进化：

    • 针对目前 PyMuPDF 对表格和多栏排版支持不佳的问题，计划切换至 SOTA 方案 Docling 或 Unstructured，实现文档的结构化高保真解析。

  • 多智能体编排：

    • Python 端将引入 LangGraph，将当前的单体逻辑拆分为 Router、Retriever、Critic 等多个 Agent，实现自我修正与反思能力。

  • 全链路可观测性：

    • 接入 Langfuse，实现对 Prompt 传输、Token 消耗和响应延迟的全链路追踪与调试。

6. 结语

Chimera-RAG 的 v0.1.0 版本是一个激动人心的开始。它证明了 “Go 工程底座 + Python AI 引擎” 这一混合范式在构建高性能、智能化应用时的巨大潜力。

未来的 v0.2.0，我们将致力于让它从一个“聪明的工具”进化为一个“专业的平台”。
