破茧重生：Chimera（v0.5.0）开发复盘

引言：在 v0.5.0 版本中，Chimera 完成了最艰难的一次“换心手术”。我们将一个单纯的知识库RAG问答工具，重构为了支持多租户、可观测、资源热插拔的企业级多智能体平台。本文将分享这次架构转型的思考与实现路径。

1. 为什么转型？（The “Why”）

在 v0.4.0 之前，Chimera 的定位很清晰：用户上传 PDF，然后提问。这解决了“单个用户对单个文档”的需求。但在实际落地企业场景时，我发现了三个痛点：

  1. 资源无法复用：销售部上传了“产品手册”，客服部想用，却得重新上传一遍。知识应该是一种底层资产，而不是依附于某个对话的附件。
  2. 黑盒焦虑：AI 回答错误时，管理员不知道是因为检索失败、Prompt 写得不好，还是模型本身的问题。我们需要可观测性。
  3. 扩展性瓶颈：今天我想接 PDF，明天想接飞书文档，后天想接 SQL 数据库。把解析逻辑写死在 Service 层是行不通的。

因此，v0.5.0 的目标很明确：平台化（Platformization）。

2. 架构重构：从功能到资源（The “How”）

2.1 核心概念的升维

我们废弃了 Document 这个狭隘的概念，引入了三个新实体：

  • **DataSource (数据源)**：一切知识的来源（文件、飞书、网页）。
  • **KnowledgeBase (知识库)**：逻辑容器，用于资源隔离。
  • **Application (应用)**：业务入口，它像一个“装配器”，将 Prompt、模型和知识库组装在一起。

这种设计让“一个知识库被十个应用复用”成为了可能。

2.2 双核通信协议的泛化

为了支持未来的 Agent（不仅仅是 RAG），我们将 gRPC 协议从具体的 ChatRequest 泛化为了 RunAgentRequest。

Go 后端不再关心 Python 到底在跑什么业务，它只负责两件事：

  1. 鉴权：你是谁？你能调这个 App 吗？
  2. 透传：将配置打包成 JSON 扔给 Python 运行时。

2.3 业务数据的闭环 (The Observability Loop)

这是 v0.5.0 最让我兴奋的特性。

以往的监控只停留在“接口通不通”。而在 Chimera v0.5.0 中，我们实现了一个完整的业务监控闭环：

  1. Python：在调用 LLM 时捕获 Token 消耗，在 LangGraph 执行时记录步骤耗时。
  2. gRPC：在流式响应的最后一帧，发送 RunSummary 消息。
  3. Go：拦截这个 Summary，结合用户信息，写入 Postgres 的 app_run_logs 表。

现在，每一次对话的成本（Token）和质量（耗时/TraceID）都被永久记录了下来。

3. 技术亮点：ETL 流水线

在 Python 端，我们引入了 Connector 设计模式。 

class BaseConnector(ABC):  
    @abstractmethod    
    def load(self) -> Iterator[DocumentChunk]:        
        pass

无论是本地文件还是未来的飞书 API，只要实现了 load 方法，就能自动接入系统的清洗、向量化（Qdrant）和图谱构建（NebulaGraph）流程。v0.5.0 内置的 FileConnector 集成了 Docling，实现了对 PDF 表格和布局的精准理解。

4. 展望 v0.6.0

v0.5.0 完成了“数据蓄水”的工作（日志入库、向量入库）。
v0.6.0 将是“数据可视化”的版本：

  • 我们将构建 Chimera Insight 监控中台，将沉淀的数据画成漂亮的图表。
  • 我们将实现 飞书/钉钉连接器，打通企业数据的孤岛。
  • 我们将实装 GraphRAG，利用 NebulaGraph 解决跨文档的复杂推理问题。
