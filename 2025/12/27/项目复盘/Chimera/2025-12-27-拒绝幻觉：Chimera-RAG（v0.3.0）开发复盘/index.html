拒绝幻觉：Chimera-RAG（v0.3.0）开发复盘

引言：RAG 的信任危机

在 v0.2.0 版本中，我搭建了一套标准的 RAG 架构（Go+Python+MinIO+Qdrant）。它跑通了，但它像个“瞎子”。 它使用 PyMuPDF 机械地提取文本，把表格读成乱码，把标题和正文混为一谈。最致命的是，当它回答问题时，用户无法验证真伪——它说“见第4页”，但我根本不知道第4页在哪里。 于是，v0.3.0 应运而生。

一、 核心突破：给 AI 装上“眼睛” (Docling Integration)

我废弃了传统的 fitz 方案，引入了 IBM 开源的 Docling。

  • 多模态解析：它不仅仅是识字，它是基于视觉模型在“看”文档。它能精准识别表格结构、段落层级和页面布局。

  • **语义切分 (HybridChunking)**：

    • Old Way: 每 500 字切一刀，经常把一句话切断。

    • New Way: 基于 Docling 的 HybridChunker，它尊重文档结构，按章节、段落切分，并且完整保留了元数据（Metadata）。

二、 体验闭环：信任源于“可溯源”

RAG 如果不能跳转原文，就是耍流氓。在 v0.3.0 中，我打通了“数据-后端-前端”的最后一公里：

  1. 数据层：Python 解析时，将 page_number 烙印在每一个 Vector Chunk 中。

  2. 交互层：

    • Prompt Engineering：强制 LLM 使用 <<filename|page>> 格式引用。

    • Vue 渲染：前端正则解析引用，将其渲染为可点击的蓝色锚点。

  3. 展示层：

    • Go 后端实现了 MinIO 文件流的 HTTP 代理。

    • 前端引入 vue-pdf-embed，实现了“点击引用 -> 右侧 PDF 自动弹出并跳转”的丝滑体验。

三、 踩坑与反思

  1. 环境地狱：Docling 的依赖极其复杂（PyTorch 等），Docker 镜像体积激增，构建时间变长。这是智能化的代价。

  2. 上下文拼接：最开始 LLM 总是无法正确引用，后来发现是因为我在 Go 端组装 Prompt 时，只拼了内容，忘了拼 Page 和 FileName。AI 只能利用你喂给它的信息，这一点至关重要。

  3. PDF 坐标系：前端 PDF 跳转一度失效，最后发现是 vue-pdf-embed 的页码属性绑定问题。

四、 下一步：迈向企业级 Agent 平台

现在的 Chimera-RAG 已经是一个极其好用的单兵武器。但我的野心不止于此。 v0.4.0 将引入 组织架构 (Organization) 和 **多智能体 (Multi-Agent)**，让它进化为支持团队协作、具备链路追踪能力的 AgentOps 平台。

代码已开源，欢迎 Star 与 PR！
