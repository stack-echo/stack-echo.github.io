[复盘]混合架构：从零构建Chimera-RAG（v0.2.0）开发复盘

前言：从“能跑”到“好用”

在 v0.1.0 版本中，Chimera-RAG 已经是一个功能完备的原型：我们使用grpc做通信，部署了 Redis、Qdrant、minio，攻克了 SSE 流式传输的跨域问题，做了一个简单的前端页面验证系统。系统跑通了，逻辑也是自洽的。

但作为一个追求长期演进的系统，v0.1.0 依然面临着“成长的烦恼”：

  • 数据孤岛：虽然有 Redis，但缺乏关系型数据库，用户数据无法持久化，文件与用户的绑定关系薄弱。

  • 同步瓶颈：Redis 虽然在场，但尚未发挥其作为“异步消息队列”的全部潜力，大文件处理依然容易阻塞主线程。

  • 代码熵增：Python 端的逻辑随着功能增加开始堆叠，急需一场架构层面的“断舍离”。

在 v0.2.0 我们对系统进行了大量的修改。我们不是在堆砌新功能，而是在深化基础设施的应用，并重塑代码的骨架。

一、 基础设施的深化应用

1. Redis：从“缓存”到“事件驱动”

在 v0.1.0 中，Redis 可能更多充当缓存或简单存储。在 v0.2.0 中，我们将其升级为 ETL 流程的神经中枢。

  • 变革：构建了标准的 Producer-Consumer（生产者-消费者） 模型。

  • 效果：Go 后端（生产者）只负责将 Upload 事件推入 task:parse_pdf 队列，耗时仅毫秒级；后台 Worker（消费者）异步拉取任务。这种解耦让系统吞吐量有了质的飞跃。

2. 引入 PostgreSQL：补全数据拼图

为了支撑多用户体系，我们引入了 PostgreSQL。

  • 价值：利用 GORM 定义了 User、Document、KnowledgeBase 模型。这标志着系统从“临时会话”走向了“持久化资产管理”。

二、 后端架构的规范化 (Go)

v0.1.0 虽然已有 Middleware 思想，但 v0.2.0 将其体系化了。

1. 鉴权体系的落地

  • JWT + DB：我们将原有的鉴权机制升级为基于数据库的 JWT 验证。

  • 中间件标准化：重写了 Gin 中间件，使其不仅负责拦截，还负责上下文注入（c.Set("userID", ...)）。这让后续的 Controller 层完全不需要关心“解析 Token”这种脏活，专注于业务。

2. Clean Architecture (整洁架构)

我们对代码进行了分层重构：

  • Data 层：收敛了所有的 DB、Redis、MinIO 操作。

  • Service 层：只关注业务编排。

  • Handler 层：只关注 HTTP 协议解析。 这种分层让单元测试和后续维护变得异常轻松。

三、 AI 内核的模块化 (Python)

这是 v0.2.0 变动最大的部分。为了迎接未来的多模态和 Agent 架构，我们废弃了单脚本模式。

  • Core 层：封装 LLM、Embedding、Splitter 等原子能力。

  • Tools 层：将 PDF 解析（PyMuPDF）独立为工具。这为 v0.3.0 替换为 Docling 埋下了伏笔。

  • Service 层：纯粹的 gRPC 胶水层。

现在的 Python 服务，已经做好了被未来 Agent 调用的准备。

四、 前端工程化 (Vue 3)

我们完善了前端的“最后一块拼图”：

  • **状态管理 (Pinia)**：持久化存储用户状态，与 v0.1.0 已有的 SSE 鉴权机制完美对接。

  • 交互闭环：新增登录/注册页，配合 Axios 拦截器，实现了用户无感知的 Token 刷新与携带。

五、 下一步：v0.3.0 预告

随着架构地基的夯实，v0.2.0 已经是一个健壮的分布式系统。接下来，我们要把精力集中在 RAG 的核心效果上。

v0.3.0 目标：AI 的觉醒

  • Docling 接入：用 AI 视觉技术替代传统的文本提取，读懂表格和布局。

  • 语义切分：基于 Markdown 标题的智能分块，不再暴力截断上下文。

  • 知识图谱：探索 NebulaGraph，向 GraphRAG 进发。

代码已开源，欢迎 Star 与 PR！
