筑牢防线：Chimera-RAG（v0.4.0）开发复盘

引言：当 Demo 遇上“现实引力”

在 v0.3.0 版本中，我引入 Docling 解决了 RAG 的“视力问题”**。但当我想把这个系统真正部署给团队使用时，原本“能跑就行”的后端架构瞬间崩塌了。

三个致命问题摆在面前：

  1. 裸奔的数据：没有组织隔离，任何人都能检索到所有向量数据。

  2. 不可测的接口：所有对话接口强制 SSE 流式输出，导致自动化测试脚本无法断言，每次改代码都要人工点点点。

  3. 脆弱的逻辑：业务逻辑耦合在 HTTP Handler 层，Service 层形同虚设。

v0.4.0 是一次深度的后端手术。我重写了 40% 的 Go 代码，并引入 Apifox 构建了完整的自动化测试防线。

一、 架构重构：从“软隔离”到“硬防御”

在 SaaS 架构中，最可怕的不是服务宕机，而是数据泄露。

1. 多租户模型的原子性重构 我废弃了扁平的用户表，构建了 User -> OrganizationMember -> Organization 的多租户模型。 最棘手的是“创建组织”的逻辑：既要创建组织，又要将创建者设为 Owner。我利用 GORM 的 Transaction（事务） 机制，确保了这两个动作的原子性——要么同时成功，要么全部回滚，杜绝了“无主组织”的脏数据。

2. 纵深防御 (Defense in Depth) 以往的 RAG 系统往往依赖向量数据库的 Filter 做隔离（软隔离）。但这不仅浪费 Embedding 算力，一旦 Filter 写错，数据即刻泄露。 我在 Go Service 层实现了 “硬隔离”：

  • 拦截机制：在调用昂贵的 Python 向量服务或 LLM 之前，Service 层强制校验 UserID 与 KbID 的所属关系。

  • 效果：非法请求在 Go 内存层即被 403 Forbidden 拦截。非法请求连向量数据库的大门都摸不到。

二、 接口工程化：驯服“流式响应”

为了让系统可测试，我必须解决 SSE (Server-Sent Events) 难以被脚本断言的问题。

我采用了一种 “双模接口” (Dual-Mode API) 设计：

  • DTO 改造：在请求体中增加 stream: bool 字段。

  • Handler 分流：

    • stream=true：走 SSE 通道，适配 Vue 前端，保持打字机体验。

    • stream=false：后端自动缓冲 Channel 数据，一次性返回标准 JSON。

  • 错误探测：在非流式模式下，如果 Service 层返回了 ERR: ⛔️，Handler 会自动将其转换为 HTTP 403 状态码。这为后续的自动化测试铺平了道路。

三、 自动化测试：Apifox 的降维打击

这是 v0.4.0 开发体验提升最大的地方。重构涉及到底层鉴权逻辑，如果靠手动测试，每一次 Commit 都是一场豪赌。

我使用 Apifox 搭建了一套包含 10+ 个步骤的自动化测试场景：

场景实战：模拟“越权攻击”

  1. 前置脚本：自动注册用户 A（研发部）和用户 B（财务部）。

  2. 数据准备：A 用户上传一份“机密文档”并建立知识库。

  3. 攻击模拟：脚本提取 B 的 Token，携带 A 的 kb_id 发起检索请求。

  4. 自动断言：Apifox 自动校验响应状态码是否为 403，以及 JSON Body 中是否包含 Access Denied。

现在，无论我如何修改底层代码，只要点击“运行”，3秒钟内我就能知道系统是否依然安全。全绿的测试报告，是重构者最大的底气。

四、 踩坑与反思

  1. 指针的代价：GORM 默认将可空字段映射为 *uint 指针。在做权限比对时，直接拿指针和 int 比较会导致 Panic。必须养成为数据库字段判空 (nil check) 再取值 (*ptr) 的肌肉记忆。

  2. 测试驱动开发 (TDD)：以前我觉得写测试是浪费时间，但这次重构证明，先写 Apifox 测试用例，再写 Go 实现代码，实际上减少了 50% 的 Debug 时间。

五、 展望 v0.5.0

地基已经打牢，防线已经筑起。 接下来的 v0.5.0，我将把目光转回 AI 本身。Chimera-RAG 目前还是个“金鱼”，只有 7 秒记忆。下一版本，我们将引入 Context & Memory，让它拥有真正的对话记忆。

代码已开源，欢迎 Star 与 PR！
